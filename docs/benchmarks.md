# SXRC Benchmarks

`sxrc-bench` is a small benchmark harness for evaluating SXRC file compression,
page compression, zram-style page classification, and `zstd`/`lz4` baselines.

## Build

```bash
cargo build --release --bin sxrc-bench
```

For offline profile generation, build the trainer too:

```bash
cargo build --release --bin sxrc-train
```

## Modes

- `--mode file`: encode/decode one whole buffer with `SxrcFileEncoder` and
  `SxrcFileDecoder`.
- `--mode ram`: split input into pages and run `SxrcRamCodec` on each page.
- `--mode zram`: simulate zram page classes:
  - all-zero page => stores zero bytes
  - same-filled page => stores one fill byte
  - otherwise use SXRC page compression, with raw-page fallback when needed
- `--mode both`: run `file` and `ram`.

## Synthetic Datasets

When `--input` is not provided, the benchmark generates one of these datasets:

- `zero`: all bytes are `0x00`
- `same`: all bytes are `0xAA`
- `repeated`: repeats a motif containing `07 FE`, `04 EE`, `90 90`,
  `48 89 D8`, and `55`
- `mixed`: alternates motif-heavy chunks with xorshift-generated bytes
- `entropy`: xorshift-generated pseudo-random bytes

## Real Corpus Input

Use `--input-dir DIR` to benchmark real page corpora. The directory must
contain `*.bin` files sorted lexicographically; each file is treated as one
page and must be at most `--page-size` bytes long.

- `--input PATH`: benchmark one flat byte stream.
- `--input-dir DIR`: benchmark a corpus of pre-split pages.

## Examples

### Mixed file + RAM benchmark

```bash
cargo run --release --bin sxrc-bench -- \
  --mode both \
  --dataset mixed \
  --size-bytes 8388608 \
  --iterations 8 \
  --page-size 4096
```

### Repeated-data zram simulation

```bash
cargo run --release --bin sxrc-bench -- \
  --mode zram \
  --dataset repeated \
  --size-bytes 8388608 \
  --iterations 8 \
  --page-size 4096
```

### Custom manifest + custom input

```bash
cargo run --release --bin sxrc-bench -- \
  --manifest ./my-profile.sxrc.yaml \
  --input ./payload.bin \
  --mode both \
  --iterations 16 \
  --page-size 4096 \
  --min-rle-units 3
```

### Real page corpus with baselines and JSON export

```bash
cargo run --release --bin sxrc-bench -- \
  --input-dir ./corpus/pages \
  --mode zram \
  --iterations 8 \
  --warmup 2 \
  --page-size 4096 \
  --baseline both \
  --latency-stats \
  --export-json ./reports/e725-zram.json
```

### Disable dynamic pattern metadata

```bash
cargo run --release --bin sxrc-bench -- \
  --mode file \
  --dataset repeated \
  --disable-dynamic-patterns
```

## Output Fields

The CLI prints one header line and one metrics line per executed mode.

Example:

```text
sxrc-bench arch=x86_64 mode=zram dataset=custom input_source=directory input_bytes=12288 iterations=8 warmup=2 page_size=4096 dynamic_patterns=true latency_stats=true baseline=both
mode=zram codec=sxrc ratio=0.180000 encode_mib_s=12.34 decode_mib_s=820.00 encoded_bytes=2211 dynamic_patterns=6 dynamic_metadata_bytes=72 encode_ms_total=12.500 decode_ms_total=0.180 encode_p50_us=37.100 encode_p95_us=51.400 encode_p99_us=60.000 decode_p50_us=0.900 decode_p95_us=1.800 decode_p99_us=2.100 zram_zero_pages=1 zram_same_pages=1 zram_sxrc_pages=1 zram_raw_pages=0
mode=zram codec=zstd ratio=0.210000 encode_mib_s=9.10 decode_mib_s=510.00 encoded_bytes=2580 dynamic_patterns=0 dynamic_metadata_bytes=0 encode_ms_total=16.900 decode_ms_total=0.290 encode_p50_us=48.500 encode_p95_us=73.700 encode_p99_us=80.000 decode_p50_us=1.200 decode_p95_us=2.400 decode_p99_us=2.800 zram_zero_pages=1 zram_same_pages=1 zram_sxrc_pages=1 zram_raw_pages=0
```

Field meanings:

- `ratio`: `encoded_bytes / input_bytes`
- `encode_mib_s`, `decode_mib_s`: throughput over all iterations
- `encoded_bytes`: encoded output size for one iteration, or the per-iteration
  average in zram mode
- `codec`: `sxrc`, `zstd`, or `lz4`
- `dynamic_patterns`: number of stream/page-local dynamic patterns used
- `dynamic_metadata_bytes`: bytes spent on dynamic pattern metadata
- `encode_p50_us`, `encode_p95_us`, `encode_p99_us`: encode latency percentiles
  in microseconds
- `decode_p50_us`, `decode_p95_us`, `decode_p99_us`: decode latency percentiles
  in microseconds
- `zram_zero_pages`, `zram_same_pages`, `zram_sxrc_pages`, `zram_raw_pages`:
  page class counts in zram mode
- `--export-json PATH`: writes the same metrics plus run metadata to JSON for
  external reporting

## Reading Results

- `ratio < 1.0` means compression gain.
- `ratio == 1.0` in RAM/zram mode usually means the page was stored raw because
  compression was not profitable.
- High `dynamic_metadata_bytes` with weak ratio can mean the dynamic pattern
  threshold is too permissive for that workload.
- Compare percentile tails, not just throughput. On legacy laptops, `decode`
  `p95`/`p99` often matter more than peak MiB/s.
- Compare `--mode file` and `--mode ram` separately. Page independence is
  better for random access, but can reduce ratio compared to whole-file coding.

## Empirical Snapshot

On synthetic 8 MiB datasets used during development:

| Dataset | Mode | Observed Ratio | Rough Saving |
| --- | --- | --- | --- |
| `repeated` | `ram` | `0.106494` | ~89.35% |
| `mixed` | `ram` | `0.974182` | ~2.58% |
| `entropy` | `ram` | `1.000000` | 0.00% |
| `zero` | `zram` | `0.000000` | ~100.00% |
| `same` | `zram` | `0.000244` | ~99.98% |

These numbers are workload and CPU dependent. Re-run on Raspberry Pi 5,
Orange Pi 5, and PS3/Linux targets before drawing platform conclusions.
